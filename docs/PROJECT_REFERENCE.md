# Project Reference

This document provides a high-level overview of the project structure and components.

## `hid` Directory

The `hid` directory is responsible for all USB Human Interface Device (HID) emulation, specifically acting as a keyboard to inject commands and run payloads on a target machine.

### Key Files:

*   **`hid_controller.py`**: This is the high-level controller for the HID system. It manages the execution of payloads, handles system state (enabling/disabling HID functionality), and provides logging. It acts as the main interface for other parts of the application that need to trigger HID attacks.

*   **`executor.py`**: This is the low-level executor that communicates directly with the underlying USB gadget device (`/dev/hidg0`). It is responsible for translating abstract commands (like "type this string" or "press CTRL+C") into the raw 8-byte HID reports that a computer understands as keyboard input. It contains the keycode mappings for characters, special keys, and modifiers.

*   **`payload_builder.py`**: This module contains the logic for building and managing the HID payloads. It defines a series of pre-configured payloads (e.g., `sysinfo`, `export_firewall`, `full_audit`) which are sequences of commands to be executed on the target. It also includes a templating system to inject dynamic data (like an IP address or timestamp) into the payloads before execution.

*   **`script.txt`**: A simple text file, likely used for testing or containing payload scripts.

*   **`upload_payload.py`**: A script to upload payloads.

*   **`hidtest.py`**: A script for testing HID functionality.

## `parser` Directory

The `parser` directory is responsible for analyzing the data files collected by the HID payloads and generating human-readable reports.

### Key Files:

*   **`audit_parser.py`**: This module parses the various output files generated by the audit payloads, such as registry exports (`.reg`), Defender status (`defender.json`), and driver lists (`drivers.txt`). It extracts meaningful data, identifies potential security misconfigurations or risks, and calculates an overall risk score.

*   **`report_generator.py`**: This script takes the structured data from the `AuditParser` and generates a professional PDF security report. This report includes a summary of findings, detailed analysis of the collected data, a risk score, and actionable recommendations.

## `portal` Directory

The `portal` directory contains the web server component responsible for receiving files from the target machine.

### Key Files:

*   **`upload_server.py`**: This script runs a lightweight HTTP server that listens for incoming POST requests. The HID payloads executed on the target machine use `Invoke-WebRequest` (PowerShell) to upload audit files (e.g., registry exports, driver lists) to this server. The server saves the uploaded files into a structured directory format based on the scan metadata (device name, timestamp, etc.) for later analysis.

## Core Application (`app.py`)

The `app.py` file is the heart of the project, running a Flask web server that provides the user interface and a REST API to control the entire system.

*   **API Endpoints**: It exposes endpoints to manage the HID system (e.g., `/hid/execute`), initiate report generation (`/report/generate`), and view system status and logs.
*   **Integration**: It initializes and integrates all the other major components:
    *   It starts the `upload_server` from the `portal` module in a background thread to listen for incoming files.
    *   It creates an instance of the `HIDController` from the `hid` module to manage and execute keyboard payloads.
    *   It calls the `generate_report` function from the `parser` module to create PDF reports from the collected data.
*   **Web Interface**: It serves the main `index.html` dashboard, which acts as the front-end user interface for the tool.

## Setup and Execution

This section describes the files related to project setup, dependency management, and execution.

### Key Files:

*   **`run.sh`**: This is the primary startup script for the application. It automates the setup process by creating a Python virtual environment, installing both system-level and Python dependencies, and finally launching the main Flask application with the necessary permissions. It also performs checks to ensure the environment is configured correctly (e.g., verifying that the HID device is present).

*   **`requirements.txt`**: This file lists the Python packages required for the project to run. It primarily includes `Flask` for the web server and `reportlab` for generating PDF reports. The `run.sh` script uses this file to install the correct dependencies.
